nanotins

A small C++20 library for parsing pcap/pcapng and L2/L3/L4 packets — sequentially or in data-parallel bulk — whose one killer idea is the overlay-able, self-describing wire struct. You describe a packet header once; nanotins hands you a zero-copy overlay, a flattened column list, an SoA store, and a nanoarrow table — so any backend (Lance, Parquet, Arrow IPC) can persist it.

C++20header-mostly deps: nanoarrow + boost::describeno Lance dependency CPU bulk today, GPU-ready

Why nanotins

Packet parsing usually means hand-written byte offsets, manual byte-swaps, and a second, totally separate body of code to get the parsed fields into a columnar table. nanotins collapses both into a single declaration. Write the struct that matches the wire; the same declaration drives parsing and the table schema. Add a protocol = add a struct + one branch. There is no schema to keep in sync, because the struct is the schema.

Overlay

Cast raw bytes to a typed view with no copy and no allocation — valid in a CUDA kernel.

Self-describing

One BOOST_DESCRIBE_STRUCT line exposes the fields to compile-time reflection.

Columnar for free

That reflection becomes an Arrow schema + an SoA store + a nanoarrow array.

Bulk & portable

The parse runs through a scheduler-agnostic bulk_for_each: CPU pool now, GPU later.

Describe a struct once

A wire struct is a normal C++ struct of nanotins field types, packed (1-aligned, no padding), with a single description line. Here is Ethernet II:

struct Ethernet {
    std::array<std::uint8_t, 6> dst;     // 6-byte MAC -> fixed_size_binary(6)
    std::array<std::uint8_t, 6> src;
    be<std::uint16_t> ethertype;             // big-endian on the wire
};
BOOST_DESCRIBE_STRUCT(Ethernet, (), (dst, src, ethertype))
static_assert(std::is_standard_layout_v<Ethernet> && sizeof(Ethernet) == 14 && alignof(Ethernet) == 1);

The static_assert is the contract that makes overlay safe: the struct is exactly the wire layout. The BOOST_DESCRIBE_STRUCT line is what lets nanotins enumerate (dst, src, ethertype) at compile time — the basis of everything columnar below.

Endianness, declared on the type: be<T> / le<T>

Wire integers carry their byte order in the type, not in scattered swap calls. be<u16> stores raw big-endian bytes; reading .host() swaps to host order on demand. le<T> is the little-endian counterpart. A plain std::uint32_t means host order (already-native fields).

be<std::uint16_t> ethertype;     // 0x0800 stored as bytes 08 00
ethertype.host();                // -> 0x0800 as a host uint16_t
Because the swap lives in the field type, a struct can mix byte orders (pcapng is little-endian framing around big-endian network headers) and every column still reads correctly.

Bitfields that straddle bytes: bits<Word, field…>

Sub-byte and byte-straddling fields are declared as a bits<> over a backing word, listing each field<"name", width> MSB-first. nanotins byteswaps the whole word first (if it is a be<>), then shifts and masks — so IPv4 flags/fragment-offset over a big-endian 16-bit word Just Work, and each named sub-field becomes its own column.

// IPv4 byte 0: version(4) | ihl(4) — one host byte, no swap
bits<std::uint8_t, field<"version", 4>, field<"ihl", 4>> ver_ihl;

// flags(3) | frag_offset(13) straddling a big-endian 16-bit word
bits<be<std::uint16_t>, field<"flags", 3>, field<"frag_offset", 13>> flags_frag;

Read the raw backing word with .word_host(); in a table, version, ihl, flags and frag_offset appear as four separate integer columns.

Zero-copy overlay

overlay(bytes, offset, out) bounds-checks and memcpys sizeof(T) bytes into a typed struct. No heap, no STL in the hot path, noexcept — the same call is valid as a __host__ __device__ kernel body. The byte order and bit layout are decoded lazily when you read a field.

Ethernet eth{};
if (overlay(packet, 0, eth)) {
    std::uint16_t et = eth.ethertype.host();   // decode on read
}

From struct to a nanoarrow table

The same description drives the columnar side. soa<T> is a struct-of-arrays store; arrow_schema<T>() emits the matching ArrowSchema (bitfields expanded to named columns, be/le as their host type, byte arrays as fixed_size_binary); and to_arrow<T>() fills an ArrowArray. That array is plain nanoarrow — hand it to any writer.

nanotins::soa<Ethernet> rows;
rows.resize(n);
for (std::size_t i = 0; i < n; ++i) rows.store(i, decoded[i]);

ArrowSchema schema{}; ArrowArray batch{}; std::string err;
nanotins::arrow_schema<Ethernet>(schema, err);   // dst, src, ethertype columns
nanotins::to_arrow<Ethernet>(rows, batch, err);   // the data
// `batch` + `schema` are now a normal nanoarrow record batch.

The two-phase pipeline (pcap/pcapng)

Capture files are parsed in two phases — the split is what makes the heavy part parallel.

PhaseFunctionShape
A — scanscan_blocks() / scan_window() Inherently sequential boundary walk. Detects pcap vs pcapng + endianness, follows the length chain, emits a flat BlockRef[] (offset, length, kind). scan_window does it over a bounded window of an endless/S3 capture, carrying section state across windows.
B — parseparse_epb() / parse_epbs_bulk() Pure, per-block, order-independent. Each BlockRef is parsed into columns with no shared state — a for loop today, a GPU kernel tomorrow (one thread per block).
std::vector<BlockRef> refs; std::string err;
scan_blocks(file_bytes, refs, err);          // Phase A: serial boundary walk
// Phase B: each ref parses independently into SoA columns (see Bulk below).

The L2/L3/L4 walk

Above the capture layer, walk_packet() is the one traversal that decodes a frame: Ethernet → VLAN* → IPv4/IPv6 → TCP/UDP. It is visited by callbacks, so the same walk powers the serial path, the bulk count pass, and the bulk scatter pass — they can never disagree.

walk_packet(link_type, pkt,
    [&](const Ethernet& e){ ... },   [&](const VlanTag& v){ ... },
    [&](const Ipv4& ip){ ... },     [&](const Ipv6& ip){ ... },
    [&](const Tcp& t){ ... },        [&](const Udp& u){ ... });
Fragmentation: the L4 header exists only in the first IPv4 fragment, so walk_packet emits TCP/UDP only when frag_offset == 0. Continuation fragments still produce an ipv4 row (every fragment carries protocol); they just have no L4 row. This matches Wireshark with reassembly off — see Validate vs tshark.

Bulk & GPU-ready

The parallel parse is a scheduler-agnostic bulk_for_each(sched, num_tasks, n, kernel) — a partitioned stdexec ex::bulk. Today the scheduler is an exec::static_thread_pool; on a CUDA host it becomes an nvexec::stream_context and the same kernel runs on the GPU. The kernel must be device-safe: POD captures, no allocation — which nanotins' overlay/store already are.

Decoding a frame emits a variable number of PDUs (0..N VLAN tags, maybe one IPv4, maybe one UDP), so the bulk path uses the canonical GPU pattern for variable output — count → prefix-sum → scatter:

// pass 1 (bulk): how many of each PDU does each packet emit?
bulk_for_each(sched, tasks, n, [=](std::size_t i){ counts[i] = count_packet(link[i], pkt(i)); });
// scan: exclusive prefix-sum per PDU type -> exact per-packet write offsets + totals
// resize each output column to its total (no over-allocation, no push_back)
// pass 2 (bulk): write each PDU to its own prefix-summed slot (disjoint, device-safe)
bulk_for_each(sched, tasks, n, [=](std::size_t i){ scatter_packet(pid(i), link[i], pkt(i), base[i], sink); });
Because the scatter writes are disjoint and ordered by packet index, the output tables are byte-identical to the serial decode — the bulk path is verified against the serial one on every test capture. On a GPU the scan becomes a thrust::exclusive_scan; the two kernels are unchanged.

Worked example: adding gPTP

Here is the entire recipe for teaching nanotins a new protocol — generalized Precision Time Protocol (IEEE 802.1AS / PTPv2), which rides directly on Ethernet (EtherType 0x88F7, no IP layer). The full module is nanotins/include/nanotins/gptp.hpp with a test in nanotins/tests/test_gptp.cpp.

Step 1 — describe the wire header

One packed struct of be<>/bits<>/array fields. reserved* members keep the byte layout exact but are left out of the description, so they never become columns — describe the fields you want, no more.

struct Gptp {                                              // 34-byte common header
    bits<std::uint8_t, field<"transport_specific",4>, field<"message_type",4>> ts_msgtype;
    bits<std::uint8_t, field<"reserved0",4>, field<"version_ptp",4>> rsv_version;
    be<std::uint16_t> message_length;
    std::uint8_t domain_number, reserved1;
    be<std::uint16_t> flags;
    be<std::uint64_t> correction_field;
    be<std::uint32_t> reserved2;
    std::array<std::uint8_t,8> clock_identity;
    be<std::uint16_t> source_port_number, sequence_id;
    std::uint8_t control_field; std::int8_t log_message_interval;
};
BOOST_DESCRIBE_STRUCT(Gptp, (), (ts_msgtype, rsv_version, message_length, domain_number,
                            flags, correction_field, clock_identity, source_port_number,
                            sequence_id, control_field, log_message_interval))
static_assert(sizeof(Gptp) == 34 && alignof(Gptp) == 1);

Step 2 — one decode branch

gPTP's discriminator is the (post-VLAN) EtherType, so its branch is a one-liner you drop into a walk right where the IPv4/IPv6 branches live:

inline bool decode_gptp(std::uint16_t ethertype, Bytes bytes, Gptp& out) noexcept {
    if (ethertype != kEtherTypePtp) return false;   // 0x88F7
    return overlay(bytes, 0, out);
}

Step 3 — you already have a table

No extra work. The struct flows through the exact same machinery as the built-in protocols, so an SoA + nanoarrow schema come for free — bitfields expand to message_type, version_ptp, … columns; clock_identity is fixed_size_binary(8):

nanotins::soa<Gptp> rows;  rows.resize(n);  /* store decoded ... */
ArrowSchema schema{}; ArrowArray batch{}; std::string err;
nanotins::arrow_schema<Gptp>(schema, err);
nanotins::to_arrow<Gptp>(rows, batch, err);     // → a gPTP table, ready for any writer
That is the whole extension: one struct, one branch, zero schema code. The test overlays a real Sync message, checks every field, then asserts the generated Arrow columns — proving the killer feature end-to-end for a protocol the core never knew about.

Validate against tshark

nanotins is paired with a dumper (nlance2table) that prints any table as CSV/NDJSON keyed by packet_id. That makes it trivial to diff a decoded table, field-for-field, against an authoritative dissector:

pcapng2lance --decode-l2l3 capture.pcapng out.lance
nlance2table -f ndjson out_ipv4.lance        # our rows, hex src/dst
tshark -r capture.pcapng -o ip.defragment:FALSE \
       -T fields -e frame.number -e ip.src -e ip.dst -e udp.dstport
Use -o ip.defragment:FALSE so tshark dissects the L4 header on the first fragment (where its bytes physically are) — matching nanotins' fragmentation gate. With reassembly on, tshark attributes UDP to the last fragment and the rows won't line up. The repo's nlance2table_tshark* tests automate exactly this comparison.

Library layout & CMake

nanotins depends only on nanoarrow + header-only boost (and stdexec for the bulk path). It knows nothing about Lance — it produces nanoarrow tables and stops there.

TargetKindWhat
nanotins::coreINTERFACE Reflection/overlay/SoA/arrow + be/le, bits, bulk_for_each. The reusable nucleus.
nanotins::pcapINTERFACE The pcap/pcapng block scanner (scan_blocks/scan_window) + per-block parse — header-only; the Phase-B parsers are device-callable, so they are inline in the header.
nanotins::protocolsINTERFACE L2/L3/L4 wire structs + the serial/bulk decode walk + the gPTP example.
nanotinsINTERFACE Umbrella — link this for the whole stack (core + pcap + protocols).
# In-tree:
add_subdirectory(nanotins)
target_link_libraries(my_app PRIVATE nanotins)        # whole stack
target_link_libraries(my_tool PRIVATE nanotins::core) # just the reflection core
