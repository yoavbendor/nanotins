nanotins architecture

A reusable C++20 library for turning raw pcap/pcapng captures into columnar Arrow tables, from one declaration per protocol. This guide explains the three ideas that make that work (wire_spec, the SoA, the DAG) and gives you a step-by-step, checkpointed recipe for adding a new parser.

The one idea to take away. You describe a protocol header once, as an explicit-offset WireSpec. That single declaration then drives everything downstream — the read, the columnar store, and the Arrow schema — and a small declarative graph says how one header leads to the next. Adding a protocol is writing one spec and one graph edge; the read path, the columns, and the tables all come for free.

What nanotins is (and isn't)

nanotins answers a narrow question well: given bytes on the wire, produce columnar tables, fast and deterministically. It does the framing & field extraction — scan the capture into packets, walk each packet's protocol layers, and lay each layer's fields out as columns. It knows nothing about Lance, files, or storage; it emits nanoarrow tables that any backend (Lance, Parquet, Arrow IPC) can persist.

Does

pcap/pcapng block scanning · L2/L3/L4 + gPTP decode · explicit wire layouts · SoA columns · Arrow schemas · a scheduler-agnostic bulk path that runs serially or on a CPU thread pool, unchanged.

Doesn't

No Lance/file I/O (that's nanolance). No stateful frame reassembly or numeric reconstruction in the core — those are higher layers. No heuristics: dispatch is exact-key only, so serial == bulk by construction.

Why it's unusual

The reader and walk are NANOTINS_HD (host+device) and allocation-free — one source of truth for the parse, with no STL in the hot path. That keeps the door open for a GPU executor layer later without a hand-ported copy.

The two libraries

nanotins is the top of a two-layer stack. Each layer depends only on the one to its left, so each can be vendored on its own.

Left→right dependency. soatins is the describe-a-struct reflection core; nanotins adds packets + the wire_spec/DAG. (A GPU/CUDA executor layer is developed separately and not part of this repo yet; the parsers are already NANOTINS_HD device-callable, so it slots on top later.)

One spec, four faces

This is the heart of the design. A WireSpec is a compile-time description of bytes on the wire. From that one declaration, four different consumers each derive what they need — and because they all read the same spec, they can never drift apart.

The spec is the shared IR; each face is a fold over its fields. The host read and the device read are the same read_field; the SoA and the Arrow schema are derived from the same field list. Change a field once and all four follow.

1 · wire_spec — the single source of truth

A WireSpec lists named fields with explicit byte offsets. Unlike a packed C struct, offsets are spelled out, so unaligned multi-byte fields and byte-crossing bitfields are first-class. Files: nanotins/wire_spec.hpp (vocab + reader), wire_spec_soa.hpp (the SoA), protocol_specs.hpp / protocol_specs_ptp.hpp (the L2/L3/L4 + PTP specs).

The three field kinds

FieldForColumn type
named_field<tag, offset, T, endian>a scalar at a byte offset (e.g. a big-endian uint16)T
named_bit_field<tag, offset, StorageT, bit_offset, width, endian> bits inside a word (e.g. IPv4 version:4); MSB-first, may cross bytes smallest uint holding width
named_bytes_field<tag, offset, N>a fixed blob (MAC, IPv4/IPv6 address) fixed-size-binary array<u8,N>

Example: the UDP header, in full

using UdpSpec = WireSpec<
    named_field<decltype("src_port"_fld), 0, std::uint16_t, wire_endian::big>,
    named_field<decltype("dst_port"_fld), 2, std::uint16_t, wire_endian::big>,
    named_field<decltype("length"_fld),   4, std::uint16_t, wire_endian::big>,
    named_field<decltype("checksum"_fld), 6, std::uint16_t, wire_endian::big>>;

// read it, anywhere (host or device):
struct_view<UdpSpec> v(p);          // p = const uint8_t* at the UDP header
uint16_t sport = v("src_port"_fld);   // byte-indexed, endian-correct, alignment-safe
Bitfields are declared by (storage word, bit offset, width). MSB-first matches RFC numbering; a field may even span two bytes of a big-endian word (IPv6 flow_label:20).

The reader is device-safe by construction

read_field<F>(p) is a tiny byte-indexed loop — no std::ranges, no misaligned word loads, no STL — marked NANOTINS_HD (__host__ __device__). So the identical read runs on the host and inside a device kernel. struct_view is just the ergonomic named-access wrapper over it.

Composing specs with embed<>

Protocols reuse sub-records (a PTP message is a 34-byte header + a per-type body; both reuse a Timestamp and a PortIdentity). embed<Prefix, Offset, SubSpec> splices a sub-spec's fields at an offset, prefixing their names — so a doubled record stays unambiguous (clock_identity vs requesting_clock_identity). It is purely compile-time: the spec still flattens to one flat field list, so the reader/SoA/Arrow paths are unchanged.

// the 10-byte PTP timestamp, reused everywhere:
using TimestampSpec = WireSpec<named_field<decltype("seconds_msb"_fld), 0, u16, big>, …>;

// a Sync body = one Timestamp at body offset 0, columns prefixed "origin_":
using PtpTimestampBodySpec = WireSpec<embed<decltype("ts_"_fld), 0, TimestampSpec>>;

2 · the SoA — wire bytes become columns

Packets arrive as an array of structs on the wire (each packet's header is contiguous bytes). A columnar table is the transpose: one contiguous array per field. wire_spec_soa.hpp derives that transpose from the spec — columns_of_spec<Spec> is the column list, and scatter writes one packet's fields across the columns.

Because each column is already a contiguous, host-order buffer, turning it into Arrow is one ArrowBufferAppend per column — and a Lance table is one step further out.
Why it matters for adding a parser: you never write column or Arrow code. Declare the spec; the SoA, the schema, and the per-column names/types are all generated from it. A bit-field becomes its own named column automatically.

3 · the DAG / FSM — how one header leads to the next

A single packet is many layers: Ethernet → maybe VLAN(s) → IPv4/IPv6 → TCP/UDP, or Ethernet → gPTP → a per-message body. Historically that's a nest of hand-written switches. nanotins expresses it as a declarative graph (spec_dag.hpp): a set of nodes, each binding a WireSpec to two pure functions, connected by exact-key edges.

Anatomy of a node

struct EthNode {
    using spec = EthernetSpec;                                  // what to parse here
    static NANOTINS_HD std::size_t advance(const uint8_t*) { return 14; }   // header length consumed
    template<class G> static NANOTINS_HD int next(const uint8_t* p) {     // where to go next
        return ethertype_dispatch<G>(struct_view<EthernetSpec>(p)("ethertype"_fld));
    }
};
  • spec — the WireSpec parsed at the current byte offset.
  • advance(p) — how many bytes this header consumes (fixed, or computed like IPv4 ihl*4).
  • next<G>(p) — read a discriminator field, look it up in a compile-time edge table, return the target node's id (or -1 = stop).
// the edge table is the additive registration point — keys are exact, never heuristic:
template<class G> NANOTINS_HD int ethertype_dispatch(uint16_t et) {
    return match_edges<G, edge<0x8100, VlanNode>, edge<0x0800, Ipv4Node>,
                                edge<0x86DD, Ipv6Node>, edge<0x88F7, GptpNode>>(et);
}
// the graph is just the ordered set of nodes; a node's id is its index:
using L2L3Graph = graph<EthNode, VlanNode, Ipv4Node, Ipv6Node, TcpNode, UdpNode, GptpNode, …>;
The L2/L3/L4 + gPTP graph. Green = leaf (the walk stops). Purple = the gPTP message-type sub-dispatch (a two-level dispatch: ethertype → Gptp → message_type → body). VLAN self-loops to handle stacked QinQ tags.

The walk — one loop, two faces

walk<Graph>(root, p, size, visit) is a bounded, allocation-free loop: parse the current node's spec, call visit(node_id, offset), advance, follow the edge, repeat. It's NANOTINS_HD, so the same loop runs on the host and inside a device kernel — the visitor decides what "emit" means (count rows, or scatter columns). Because both the count pass and the scatter pass use the same walk, they can never disagree.

Reject heuristics — this is a hard rule. Edges match an exact key (an ethertype, a protocol number, a message type). There is no "looks like…" guessing. That is what guarantees serial == bulk bit-for-bit and what makes adding a protocol additive (a new edge can't change how existing packets parse). A duplicate key is a build-time error, not a silent ambiguity.

4 · bulk decode — count → scan → scatter

A packet emits a variable number of rows per table (0 or 1 Ethernet, 0–N VLANs, 0/1 UDP…). The data-parallel pattern for that is three passes (dag_bulk.hpp):

The scan is a host loop and the passes run through bulk_for_each — serially or on an exec::static_thread_pool. The scatter sink is a plain POD of pointers (trivially copyable — which is also what would let a device kernel capture it), and the parse is the same walk. Result: the output tables are byte-identical whether produced serially or on a thread pool. (Because the sink and kernels are POD + NANOTINS_HD, the same three passes are what a future GPU executor would drive via a device scan — no change to this code.)

End-to-end pipeline (the pcapng2lance example)

capture.pcapng
   │  pcap_blocks: scan a bounded window → BlockRef[]            (streaming, RAM-bounded)
   ▼
 EPB packets ──► L1 SoA columns + a lance.blob.v2 payload_ref    (payload stays external, never copied)
   │
   ▼  --decode-l2l3
 walk<L2L3Graph> per packet  ──►  per-node dag_tables<Graph>     (eth/vlan/ipv4/ipv6/tcp/udp/gptp/…)
   │                                   serial · CPU pool — identical output
   ▼
 to_arrow per node  ──►  one Lance table per PDU type

The packet_id column ties every PDU row back to its L1 packet (the join key). Each per-node table is written to its own Lance dataset, byte-identical regardless of executor.

Adding a new parser — the recipe

The pattern is always the same: spec → node → edge → (table) → verify. Each step has a cheap checkpoint so you never debug five things at once. Below, each step names the file, the change, and the test that proves it before you move on.

1

Write the WireSpec

In nanotins/include/nanotins/protocol_specs*.hpp, declare the header with explicit offsets/endianness. Reuse sub-records with embed<>. Field names become column names, so choose them deliberately.

Checkpoint A — oracle read. Add a unit test (model: nanotins/tests/test_wire_spec_*.cpp) that builds deterministic bytes and asserts struct_view<YourSpec>(p)("field"_fld) equals the value computed by hand (or by the existing protocols:: overlay / soatins bits<>). This proves offsets, endianness, and bit math before any graph wiring.

2

Add a DAG node

In spec_dag.hpp: forward-declare the node, then define it with using spec =, an advance() (fixed or computed), and a next<G>(). If it's a terminal protocol, next returns -1 (a leaf). Append the node type to the graph alias.

Checkpoint B — routing. Extend nanotins/tests/test_spec_dag.cpp: craft a packet that should reach your node and assert the walk's visited sequence is […, YourNode@offset]. For an L2/L3/L4 protocol, the test also cross-checks the walk against the hand-written walk_packet for parity.

3

Wire the edge from the parent

Add one edge<KEY, YourNode> row to the parent's dispatch (e.g. a new ip_proto goes in ip_proto_dispatch; a new ethertype in ethertype_dispatch; a new PTP message in ptp_msgtype_dispatch). The key must be exact and unique — siblings are untouched, which is the whole point.

Checkpoint B still passes, and the existing L2/L3/L4 parity cases are unchanged — proof your edge didn't perturb any other packet.
4

Emit the table (driver)

In the example driver (pcapng2lance_main.cpp, write_pdu_tables), add one write_one("_yourpdu.lance", std::get<node_id_v<YourNode, G>>(dag_pdus_)). The table schema + columns are already generated from the spec — no Arrow code.

Checkpoint C — byte-identical & correct. Two independent checks: test_pdu_table_interop (the DAG table equals the reference path's Arrow buffers), and a tshark alignment test (model: test_nlance2table_tshark_ptp.py) that diffs every column against Wireshark's dissection of a real capture. This is the "is the parse actually right?" gate.

5

Bulk — nothing to write, just verify

The bulk path (dag_bulk.hpp) is generic over the graph — it reuses the same count_packet_dag / scatter_packet_dag kernels, so your new node needs no bulk code.

Checkpoint D — determinism. test_dag_bulk asserts serial == bulk bit-for-bit (count→scan→scatter equals the sequential decode). (The same harness gains a == gpu leg when the GPU executor layer is re-introduced.)
That's the whole loop. One spec, one node, one edge, one table line — and four checkpoints that each isolate exactly one kind of mistake (read math → routing → output equality → executor determinism). You never touch column code or Arrow code.

Worked example: a new L4 protocol (SCTP)

// 1) protocol_specs.hpp — the 12-byte SCTP common header
using SctpSpec = WireSpec<
    named_field<decltype("src_port"_fld),     0, u16, big>,
    named_field<decltype("dst_port"_fld),     2, u16, big>,
    named_field<decltype("verification_tag"_fld), 4, u32, big>,
    named_field<decltype("checksum"_fld),     8, u32, big>>;

// 2) spec_dag.hpp — a leaf node
struct SctpNode {
    using spec = SctpSpec;
    static NANOTINS_HD std::size_t advance(const uint8_t*) { return 12; }
    template<class G> static NANOTINS_HD int next(const uint8_t*) { return -1; }
};
using L2L3Graph = graph<…, UdpNode, SctpNode>;            // append it

// 3) the one edge: SCTP is IP protocol 132
return match_edges<G, edge<6, TcpNode>, edge<17, UdpNode>, edge<132, SctpNode>>(proto);

// 4) driver: write_one("_sctp.lance", std::get<node_id_v<SctpNode, G>>(dag_pdus_));

Total: ~12 lines across three files, plus tests. No column or Arrow changes. The sctp.lance table — columns packet_id, src_port, dst_port, verification_tag, checksum — is byte-identical whether decoded serially or in bulk.

A protocol with sub-types (like gPTP's message bodies) is the same recipe one level deeper: the node's next dispatches on a discriminator field into a second set of body nodes — see GptpNode → ptp_msgtype_dispatch → {PtpTimestampBody, PtpTsPortBody, PtpAnnounceBody, PtpSignalingBody}.

When your parser needs new dependencies

The core wire_spec/DAG path is deliberately header-only and device-safe — no STL containers in the hot path, no allocation, no third-party headers that aren't device-clean. Keep it that way. Where a new dependency lives depends on what it is:

Your parser needs…Put it…Because
just more wire fields / a new layerin nanotins (spec + node), header-only stays device-safe; bulk-parallel for free
cross-packet state (reassembly, PTP offset correlation, sensor frame assembly) a higher layer above nanotins (host-only), consuming the per-node tables by packet_idstateful logic isn't data-parallel; it must not pollute the device walk
heavy numeric reconstruction (floating-point, calibration)a separate compute stage on the columnar outputfloating-point determinism is its own battle; isolate it
a third-party lib (decompression, crypto…)behind an interface in a consumer (e.g. the driver), never in the NANOTINS_HD pathkeeps nanotins vendorable + device-safe

Mechanically: a new header-only dep is a target_link_libraries(nanotins_protocols INTERFACE …) or a FetchContent block in nanotins/CMakeLists.txt; a host-only dep belongs to the consumer.

The three layers (where things belong)

layer 1

Framing & dispatch

wire_spec + spec_dag. Stateless, device-safe, exact-key. This is nanotins. Adding a protocol normally lives entirely here.

layer 2

Stateful assembly

Correlate/Reassemble across packets (PTP sync↔follow-up, fragmentation, sensor frames). Host-only, consumes layer-1 tables. Lives above nanotins.

layer 3

Numeric reconstruction

Turn raw fields into physical values. Separate compute; this is where floating-point determinism is fought, kept away from the parse.

Rule of thumb: if it can be a pure function of one PDU's bytes, it's a layer-1 spec field. If it needs to remember another packet, it's layer 2. If it produces a number a physicist cares about, it's layer 3.

Rules & gotchas

Device-safety is contagious — protect it. Anything reachable from a node's advance/next or the walk visitor must be NANOTINS_HD and allocation-free. No std::string, no std::vector, no exceptions in that path. The host-side table assembly (which does use std::vector) is separate from the device walk.
  • Exact keys only. Never dispatch on a range or a guess. A duplicate edge key is a compile error, by design.
  • Bitfields are MSB-first and declared by (storage word, bit offset, width). A field may cross bytes inside its big-endian word.
  • Names are the schema. A field's tag is its column name; an embed<> prefix disambiguates repeated sub-records. Pick names you'd want in a Lance table.
  • g++ fold gotcha: don't nest a lambda inside a fold-comma expression (it ICEs older g++); factor per-index work into a helper, as dag_bulk.hpp does.
